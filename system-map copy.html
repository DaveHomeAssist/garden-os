Garden OS
System Map v2

A fuller map of the repo as it actually exists now: root static tools, a built story-mode/ app, shared canon in specs/ and docs/, and Let It Grow as a planned mode rather than a live runtime.

Garden OS is no longer one single architectural line. The repo currently combines a root HTML tool suite, a legacy simulator lineage, Story Mode as the active flagship runtime, and an expanding canon across specs/, root docs/, and docs/story-mode/. This draft reframes the map around that hybrid state and the storage rule that matters now: resources should be stored by scope and lifecycle, not by version label alone.

01

Current State Snapshot

The repo has four practical lines in play at once. This page treats them as one coordinated system with clear source-of-truth boundaries instead of flattening everything into one planner-centric stack.

⬡
Hybrid Repo
Garden OS now combines root-level static HTML surfaces, shared docs and specs, generated output trees, and a separate built application in story-mode/. The repo cannot be described honestly as single-file-only anymore.
▣
Story Mode Flagship
story-mode/ is the current flagship runtime. It owns the active game loop, built app packaging, Three.js scene layer, and save flow, while the root surfaces still serve planning, explainability, and legacy play support.
◈
Shared Canon
The current order of truth runs through specs/, docs/HANDOFF.md, root docs/, and then docs/story-mode/. That canon layer is more important than any individual page or generated artifact.
⟁
Let It Grow In Scope
Let It Grow is in scope as planning and future mode design inside docs/story-mode/. It is not a separate deployed runtime, not a separate repo line, and not yet a peer to Story Mode in executable code.

04

Canonical Source Hierarchy

The repo now has enough overlapping documentation that the page needs to show source precedence explicitly. This is the order contributors should reason from when pages, docs, or experiments disagree.

01
specs/
Shared gameplay contracts, scoring rules, schemas, and mode-shaping canon belong here first. If a rule is shared across active lines, this is the strongest place to anchor it.
02
docs/HANDOFF.md
The handoff document is the clearest current-state summary for the repo as a whole. It describes the hybrid architecture, current live surfaces, and current tension between root tools and Story Mode.
03
root docs/
Repo-wide SOPs, architecture notes, launch material, URL registries, and storage/versioning policy live here. This is the main repo canon layer outside strict rules and schema files.
04
docs/story-mode/
Story Mode planning, Let It Grow expansion framing, world design, systems docs, and technical notes live here. These are active mode docs, but they do not automatically outrank shared repo canon.
05
progress logs and notes
Files like progress.md, implementation notes, and roadmap traces are useful for context and chronology, but they should not silently override specs or core docs.
06
generated and archived artifacts
dist/, output/, reports/, and archive trees are downstream artifacts. They matter operationally, but they should not become the place people edit from first.
05

Shared Resource Boundaries

Storage needs to follow ownership now. The real problem is no longer version count by itself. The real problem is mixing shared canon, runtime-local assets, generated output, and archived files in the same mental bucket.

◉
Shared Canon Lives Once
If a rule or data set means the same thing across multiple active lines, it should live once in specs/ or a future shared data namespace such as data/shared/. Duplicate canon is where version sprawl starts.
◌
Runtime-Local Stays Local
Story Mode-only assets and authored game data belong in story-mode/assets/ and story-mode/src/data/. Future planner-only or legacy-only resources should follow the same rule instead of landing at repo root.
◎
Generated Is Not Source
dist/, story-mode/dist/, output/, and similar trees may be necessary for deployment or validation, but they are generated artifacts. They should stay labeled and downstream.
◇
Current Cleanup Targets
The biggest current cleanup targets are loose root JSON files such as achievement-registry.json and crop-roster.json, plus the double archive pattern of archive/ and archives/.
06

State, Persistence, and Delivery

Root tools and Story Mode use different packaging models, but they still share the same repo and the same delivery surface. This section names how state moves and where the deploy boundary actually sits.

01
Root tools persist locally
Planner and root support tools store user state in the browser. They still operate as static pages with local persistence rather than server-backed applications.
02
Portable .gos.json flow
Export and import remain part of the root tool contract. gos-schema.json and related spec material anchor that portability instead of leaving it implicit in page code alone.
03
Story Mode keeps its own save state
Story Mode manages campaign and gameplay state in its own app model. It shares the repo, but it is not packaged or organized like the root single-file tools.
04
GitHub Pages serves both layers
The root project pages and the Story Mode branch live under the same public hosting umbrella. Delivery is unified at the site level even though the runtime architectures differ underneath.
05
Generated delivery artifacts stay labeled
dist/, story-mode/dist/, output/, and reports/ need to remain clearly downstream so contributors do not confuse deploy products and validation captures with authored source.
07

Current Risks and Future Boundaries

The repo is workable, but the friction points are visible. This draft names them directly so the map does not accidentally flatten current architectural tensions into a cleaner picture than reality.

R1
Root Rules No Longer Cover Everything
Older repo assumptions were written for a root-level single-file tool suite. They do not fully describe the built Story Mode branch or its runtime-local asset and data model.
R2
Resource Sprawl By Version
Several files are still effectively organized by history and convenience rather than by scope and lifecycle. That makes it too easy for each new branch to create its own data and asset silo.
R3
Legacy Simulator Overlap
The legacy season simulator lineage and Story Mode overlap in mechanics and campaign ideas. That overlap is useful context, but it is also a source of duplicated concepts and naming drift.
R4
Let It Grow Is Planned, Not Runtime
Let It Grow belongs in scope as future mode design and product framing, but the repo should not pretend it already has executable parity with Story Mode. Its current home is documentation, not runtime code.
R5
Shared-Core Is Future, Not Present
A shared engine or shared-core extraction may make sense later, but it is still a boundary to define, not an implemented architecture to advertise. Resource cleanup comes before shared-core storytelling.
R6
Archive Discipline Matters
Historical HTML, reports, output captures, and experiments need one clearly marked archive path. Otherwise active work and frozen history keep competing for the same top-level visual weight.
