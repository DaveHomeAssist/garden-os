==============================================================================
1. PROJECT SUMMARY
==============================================================================

OBJECTIVE
- Ship a combined v4.5 release that turns garden-planner-v4.html from a
  static plan surface into an active daily coach. Headline moment: user
  opens the planner, sees 3 to 5 concrete tasks for today, derived from
  cell state plus a live local weather forecast.

KEY OUTCOMES
- Inline weather module fetches a 7-day forecast from Open-Meteo
  (free, CORS, no key) with a 6-hour localStorage cache and offline
  fallback.
- Inline `buildTodayTasks(state, weather, now)` returns up to 5 ranked
  tasks. Pure function; deterministic given inputs.
- `today` accordion row renders a weather strip, a task list with
  mark-done and snooze, and a top-of-page frost banner.
- Frost-sensitive crops get a per-cell ghost overlay on frost nights,
  after the canonical `frostSensitive` patch lands (Phase 4 in this plan).
- Optional local notification opt-in for frost alerts (no push server,
  no backend).
- Tests exercise `buildTodayTasks` with fixture state plus fixture
  weather JSON.
- Everything inline. Zero backend, zero new external deps,
  offline-capable except for the explicit Open-Meteo call.

PHASE 1 — Weather module, author and test in isolation
--------------------------------------------------------------------------
Goal: working, tested weather module as a standalone script, NOT yet
inlined into the planner.

Key milestones
- Author `tools/weather.dev.js` (dev-only, not part of shipped root
  surface) exposing:
    weather.fetchForecast()    -> Promise<WeatherSnapshot | null>
    weather.isFrostTonight(s, frostThresholdF) -> boolean
    weather.heatIndexNext24(s) -> number
    weather.rainProbNext24(s)  -> number
- Open-Meteo endpoint:
    https://api.open-meteo.com/v1/forecast
      ?latitude=<lat>&longitude=<lon>
      &daily=temperature_2m_min,temperature_2m_max,precipitation_probability_max
      &hourly=temperature_2m,precipitation_probability
      &timezone=auto
- Cache: localStorage key `gardenOS_weather_v1`, 6-hour TTL,
  stale-while-revalidate.
- Two committed fixtures under `tests/fixtures/weather/`:
  `frost-tonight.json`, `summer-heat.json`.
- Simple test harness at `tests/weather.test.html` that loads the
  fixtures and asserts boolean and number outputs.
- Error contract: no location or fetch failure returns `null`.

Dependencies
- Phase 0 complete. Does NOT require Phase 5B to be merged.

Required resources
- Open-Meteo public APIs (no key).

Estimated timeline: 1 day.

Owner: implementer.

Risks and failure points
- CORS regression on older Safari. Mitigation: iOS 16 and 17 smoke.
- Open-Meteo rate limits. Mitigation: 6h cache, single request per load.

PHASE 2 — Today engine, author and test in isolation
--------------------------------------------------------------------------
Goal: pure function `buildTodayTasks` produced and tested against
fixtures, NOT yet inlined into the planner.

Key milestones
- Task schema:
    { id, cellId, crop, action, priority, reason, factors[], dueWindow }
- Action enum v1: `water`, `check`, `harvest`, `cover_frost`,
  `shade_heat`, `succession_plant`, `thin`, `trellis_check`.
- Inline `TASK_RULES` as a JSON literal keyed by
  `(cropFaction, cropStage, weatherSignal) -> candidateAction`.
  v1 coverage: 8 factions x 3 stages (sprout, growing, harvest-ready)
  x 4 weather signals (frostTonight, heatHigh24, rainHigh24, calm).
  Treat `frostSensitive` as `null` for now; Phase 4 promotes it once
  the data patch lands.
- `buildTodayTasks(state, weather, now)`:
    * Iterate cells with a crop.
    * Derive stage from `plantedAt` and `daysToMaturity`.
    * Evaluate rules, sort by priority, cap at 5.
    * If `plantedAt` is absent, estimate from workspace creation time
      and tag the task reason with `estimated=true`.
- Persistence shape drafted (but not yet written): `gardenOS_tasks`
  keyed by `YYYY-MM-DD:taskId`.
- Standalone harness `tests/today.test.html` covers 5 fixtures: empty
  bed, all-frost, summer heat, mid-harvest, succession-ready.

Dependencies
- Phase 1 complete OR a weather fixture stub in place.
- Does NOT require Phase 5B to be merged.

Required resources
- `CROP_SCORING_DATA.json` read access (canonical). Do NOT patch it
  here; that is Phase 4 scope.

Estimated timeline: 1 to 1.5 days.

Owner: implementer.

Risks and failure points
- `plantedAt` missing on older workspaces. Mitigation: graceful
  estimate plus clear reason text.

PHASE 5 — Optional local notifications for frost alerts
--------------------------------------------------------------------------
Goal: user receives a frost warning even when the planner tab is
backgrounded, without any push server.

Key milestones
- Settings toggle "Notify me on frost". Calls
  `Notification.requestPermission()` only on explicit user tap.
- Register service worker `periodicsync` where available
  (Chrome Android): daily 6pm local check reads the cached forecast
  and fires a local notification if `isFrostTonight` is true.
- Fallback where `periodicsync` is unavailable: on-open check between
  4pm and 10pm local; if permission is granted and frost is forecast,
  fire an in-tab notification.
- Deduplicate per calendar date via
  `gardenOS_notify_last_date` in localStorage.
- UI reflects live permission state on each load.

Dependencies
- Phase 4 complete.

Required resources
- Existing `sw.js`.

Estimated timeline: 1 day.

Owner: implementer.

Risks and failure points
- iOS Safari lacks `periodicsync`. Mitigation: on-open fallback.
- Permission revocation mid-session. Mitigation: re-check per load.

3. EXECUTION NOTES
==============================================================================

CRITICAL SEQUENCING CONSTRAINTS
- Phases 1 and 2 DO NOT touch `garden-planner-v4.html`. They author
  and test inside `tools/` and `tests/` only. They can proceed even
  while Phase 5B is in flight.
- Phase 3 REQUIRES IMPLEMENTATION_PLAN.md Phase 5B to be merged. Block
  and wait if not.
- Phase 4 REQUIRES IMPLEMENTATION_PLAN.md Phase 4 and Phase 5 reasoning
  track to be fully closed before any `specs/CROP_SCORING_DATA.json`
  edit, per the "do not extend stored crop schema in Phase 4" gate.
- Phase 5 only after Phase 4 because permission gating depends on
  final UI paths.
- Phase 6 last; no scope creep.

PARALLELIZABLE WORK
- Phases 1 and 2 can run in parallel if two implementers exist, once
  Phase 0 is done.
- Test fixture authoring for Phase 6 can begin during Phase 1.
- Documentation drafts for Phase 6 can begin during Phase 3.

QUICK WINS
- Standalone weather module plus fixtures (Phase 1) gives a testable
  artifact in one day without risking any merge conflict.
- Standalone today engine fixtures (Phase 2) prove the rule set
  before integration.

LONG-LEAD TASKS
- Rule set coverage across all 8 factions and all stages.
- `frostSensitive` per-crop classification call (Phase 4) needs human
  input for the borderline crops.
- Cross-browser notification behavior (iOS Safari quirks) in Phase 5.

NON-GOALS FOR v4.5
- Photo harvest log (Feature 3). Deferred.
- Visitor mode share link (Feature 4). Deferred.
- Cross-season memory overlay (Feature 5). Deferred.
- Server push notifications. Prohibited by zero-backend rule.
- Any change under `story-mode/`. Separate track.
- Any change to scoring numbers, total calculations, or reasoning
  path. Phase 5B owns that space through 2026-04-24.

HARD CONSTRAINTS (from CLAUDE.md, not negotiable)
- No backend, servers, APIs, or databases.
- Root tools stay single-file HTML. No imports, no modules, no bundler
  at the root.
- Offline-capable except for the explicit Open-Meteo fetch.
- Specs override code. When `specs/CROP_SCORING_DATA.json` changes
  (Phase 4), every consumer updates in lockstep.
- No randomness in scoring. This plan does not touch scoring.

SHIP GATES (from IMPLEMENTATION_PLAN.md reasoning track)
- Do not extend the stored crop or cell schema in Phase 4.
- Do not change scoring totals while reasoning is still being
  separated from the UI layer.
- Do not ship a save migration in the same pass as a new reasoning
  contract unless rollback has been rehearsed.
- Keep planner-track work additive to the existing planner surface.
